The problem

Every conversation starts at zero.

The most expensive thing about working with an AI agent today is not the inference cost. It's the cost of re-establishing context every single conversation.

Who are you. What you're working on. Who's on your team. What time you're available. What you refuse to do. What you already tried. What you learned from that. What you decided last Friday.

Without a place to put it, this context lives in your head — and dies the moment a conversation ends. Each new session, the agent starts blind again.

A notes app doesn't fix this. A notes app is for you. The agent can't read it, can't reason over it, can't tell what's still true. The problem isn't storage. It's structure.

Every session, the same warm-up. Friction tax on a person who already explained this last Tuesday.
With PCA

Claude: [reads get_self_summary]

The agent already knows the constraints, collaborators, and stances. The conversation starts where it should: the actual question.

An anatomy of context

Twelve layers, six categories.

PCA decomposes "what an agent should know about you" into twelve typed layers, grouped into six categories. Each layer has its own shape, its own validation, and its own epistemic decay rate — because moods change in days, goals shift in months, and identity changes in years.

Each layer has a typed schema, a default time-to-live, and its own SQL view. Twelve cleanly separated facets of "what an agent should know" — no overflow, no confusion.

Context becomes a graph

Eleven canonical relations.

Twelve layers of facts would still be a flat list. What makes PCA different is that the facts connect. Each link is directed, typed against an allow-list, and validated for cycles and pair compatibility. The vocabulary is deliberately small — eleven relations cover every structural connection an agent needs.

Six edges turn seven facts into a queryable structure. Ask the graph: "what does my exam goal depend on?" — and the answer is deterministic, not a guess.

Acyclic relations (today: only subgoal-of) are enforced with BFS cycle detection on link creation. Pair validation rejects nonsense links before they exist in the graph.

Capture in seconds. Used forever.

One skill. One pipeline.

You don't open an app. You don't click through forms. You type /ctx-add in Claude Code, in whatever language feels natural, and the skill does the work.

The capture pipeline

  1. Read before write. Skill calls get_self_summary and get_relevant_context to catch duplicates and resolve vague references ("the goal", "Razvan", "last week").
  2. Classify into one of twelve. Decision rules tell stance from preference, state from event, goal from role.
  3. Propose splits when needed. "Mihai (son, 18) prepares for UMF Carol Davila, July 2027" becomes a person + event + place + two links.
  4. Input fidelity. Persist only what is literally in your input. Any agent-side inference becomes a separate entity flagged inferred — never quietly merged into a self-declared fact.
  5. Confirm before persist. You see the proposed entities and links and answer [Y / n / edit / split].
  6. Persist + audit. Every write logs an event row. Nothing is silently mutated; nothing is silently lost.

What you actually type

# single fact — auto-typed
/ctx-add I refuse to build dashboards for retail SaaS — too crowded

# multi-fact — proposes splits
/ctx-add Razvan (cofounder, TS expert) joined PCA this week

# a transient state — auto-decays in 7 days
/ctx-add Today I'm focused but low energy

# Romanian works — skill is language-agnostic
/ctx-add Lucrez la spital luni–vineri 8–15, hard constraint
The "input fidelity" rule, in plain language.

The skill is allowed to interpret what you say, but never to inject facts you didn't say. If the agent thinks "because you said X, you probably also believe Y", then Y goes in the store as a separate entity with authority: inferred — never quietly merged into your self-declared profile. This is what keeps the store from drifting into LLM hallucination of you.

Where did this fact come from?

Every entity carries its provenance.

A memory store is only as useful as it is trustworthy. PCA tags every entity and every link with an authority field that records how that fact got into the store. Three values, three trust levels.

Agents read this field. A well-built skill follows your self-declared constraints strictly but treats inferred ones as hypotheses to be checked. When the agent gets something wrong, you can audit the events log and see exactly where the wrong fact came from — and how trusted it was.

No generic memory tool does this. Mem, Reflect, Notion, Obsidian — they treat every note as equally true. PCA treats provenance as a first-class column, because agents need to know how much to trust their own memory of you.

Facts have expiration dates.

The store stays current by construction.

The reason most personal-knowledge systems become useless after six months is the same: they accumulate without revisiting. Yesterday's truth becomes today's noise.

PCA gives every entity a time-to-live that matches its epistemic decay rate. A mood lasts a week; a goal lasts a quarter; a subscription, half a year; a stance, until you change your mind.

When something expires it doesn't disappear — it goes stale. Stale entities surface in ctx review. You confirm with one of three outcomes, and the system updates accordingly. If you invalidate the entity, every link that touched it is automatically invalidated too — no orphaned references, no zombie facts.

Whoever you are, your context fits.

The twelve layers are categories of personhood — not domain objects.

The schema isn't shaped around developers, founders, or any specific job. It's shaped around how a human relates to the world — what you want, what blocks you, who matters, how you think. That's why it generalizes. Pick a persona and see.

Why this and not a notes app

Five things that make PCA different.

Any one of these can be added to a memory tool as a feature. The five together compose a system — each piece reinforces the others, and that's what makes the combination hard to copy.

— 01

Agent-native, not user-native.

You don't open the PCA app, because there isn't one. Capture lives inside Claude Code via /ctx-add. The agent reads the store at session start. The whole loop runs inside your flow — zero context switch.

A note in Mem requires you to open Mem. A fact in PCA is captured while you're already talking to the agent that needs it.
— 02

Structural reasoning over a typed graph.

Twelve typed layers, eleven validated relations. The agent doesn't grep your notes — it traverses a structure. "What goals require a resource I just invalidated?" is a SQL query, not a guess.

At fifty entities you can read everything. At five thousand, only the graph-shape can answer "show me the dependency chain from this goal."
— 03

Provenance you can audit.

Every entity carries an authority: self-declared, observed, or inferred. Every change writes an event row. When the agent gets you wrong, you can trace exactly which fact misled it.

Most memory tools store all notes as equally true. PCA's events log is forensic-quality — and that's what builds trust.
— 04

Multi-horizon scopes.

You aren't one person across all contexts. Entities carry a scope field — role:hospital, project:alteramens, persona:family. Agent behavior can shift implicitly with active scope.

Same person, different operating mode. Generic tools force you to manually "set context" every time. PCA bakes it into the data.
— 05

Continuous alignment by construction.

TTL on every layer matches its decay rate. Stale entities surface in ctx review. Weekly confirmation extends, modifies, or invalidates — and invalidations cascade through the graph.

Set-and-forget memory rots. PCA forces a small weekly reckoning — a ritual that strengthens self-knowledge, not just storage.
— and

The compounding property.

Each angle alone is replicable. The combination is a system: typed graph only works because vocab is canonical; canonical vocab only stays clean because of input fidelity; input fidelity only matters because agents read provenance to calibrate trust.

A defensible product isn't one feature — it's how the pieces hold each other up.

The agent's hands and eyes

Nine MCP tools, one shaped surface.

Everything an agent does with PCA happens through nine MCP tools. They split cleanly into read, write, and link. Each is zod-validated, audit-logged, and designed to be called blindly by an LLM — descriptions explain not just what they do but when to call them.

ToolGroupPurpose
get_self_summary read Identity snapshot. Self + active roles · goals · constraints · recent state · top people · stances · preferences · key_links. Cheap (~10 ms), safe at session start, designed as the first call any agent makes.
get_relevant_context read FTS5 full-text search over title + body, optional type filter. Returns ranked entities. For ad-hoc lookups when the agent has a specific query.
list_active read All active and unexpired entities of a given type. Persons ordered by importance; everything else by recency.
get_neighbors read Direct link-graph traversal from a center entity. Returns each neighbor with its link metadata and direction (out/in).
record_observation write Create entity. Auto-types as event if no type given. Applies default TTL unless explicitly null. Returns id + applied TTL.
update_entity write Partial update by id. Returns {previous, current} so the caller can diff.
confirm_entity write Resolve a stale entity: still-true (extends TTL) · no-longer-true (invalidates) · modify (updates + extends).
link_entities link Create typed link between two active entities. Validates vocab + allowed pair + cycles (for acyclic relations).
invalidate_link link Append-only link revocation. The link row is never deleted — just stamped with invalidated_at. Events log records the cause.
Transport: stdio. Validation: zod on every input. Errors arrive at the client as structured envelopes (HandlerError with a code), never as transport crashes.

Later — only if arc 2 gate passes

03 Hosted, multi-tenant.

  • Tenancy in the schema. tenant_id on every table. Per-tenant FTS index or sharded SQLite files (cleanest).
  • Identity binding to MCP. Recommended path: local stdio + libSQL embedded replica that syncs to cloud. Operator stays offline-capable; cloud is durable backup + cross-device sync.
  • Sync semantics. Minimum: LWW on entities, additive on links, invalidation wins. Real: events log is source of truth, derived state replays.
  • Billing primitives. Per-seat (simple) or per-entity-tier (aligned with value). Meter via the events log.
  • Operational surface. Per-tenant logs, p95 latency, backups, consent-gated support inspection.
  • Gate: 10 paying customers from arc 2 + 3 betas. Below that bar, do not ship hosted.

How it's built

Small parts. Sharp boundaries.

A monorepo of four packages. Each has a clear job; each is independently testable. The whole thing runs on Bun + TypeScript + SQLite. No services to manage, no API keys to rotate.

/ctx-add skillclassifier + capture UX · ~396 lines
pca CLI · ctx CLIsetup, doctor, link hygiene
@pca/mcp-server9 tools · zod inputs · stdio
@pca/corestore, entity registry, link relations
~/.pca/store.dbSQLite · WAL · 16 tables · FTS5 · 14 active views

What's in each package

core — the engine. Schema migrations, the 12 entity registry (zod-validated attrs + TTL), 11 canonical relations (pair validation + cycle detection), the store API (~830 lines).

mcp-server — wraps core in 9 MCP tools. computeKeyLinks() is the brain behind get_self_summary — ranks high-signal links touching identity anchors.

cli — two binaries. pca is admin (init, doctor, install). ctx is runtime (link hygiene review).

skill-ctx-add — the only thing the LLM actually loads. The capture contract. ~396 lines that turn free text into structured entities and links.

"Make every part as small as it can be, and the boundaries as clean as you can. Then the next person to touch it — including future you — has a fighting chance." — the design north star, in one sentence
