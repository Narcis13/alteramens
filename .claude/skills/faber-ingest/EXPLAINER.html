WIKI_ROOT=$(d="$PWD"; while [ "$d" != "/" ]; do
  [ -f "$d/wiki/.faber.toml" ] && { echo "$d/wiki"; break; }

02Probes · the queries this skill runs

Read the index, before you change it.

The skill is mostly reads. It interrogates faber.db to avoid duplication, to find related pages, to surface active self-context, and finally — once — to rebuild the index after writing.

Duplicate detection · before ingest

GET pages WHERE type='source' AND source_ref LIKE :kw Find existing source pages by URL fragment or title keyword. read
GET aliases WHERE alias LIKE :name Catch entity duplicates hiding behind a different spelling. read

Self-context · before Phase 1

GET v_self_active_context Pull active pillars_json and stances_json as alignment yardstick. read
GET self_pillars WHERE status='active' Resolve pillar slugs for the alignment: frontmatter in Phase 3b. read

Verify · after writing, before reporting

GET v_phantoms Quick lint — surface any wikilink targets that don't yet exist as pages. read
GET v_narcis_alignment GROUP BY pillar, relation Confirm the alignment write took, broken down by pillar and relation. read

Mutate · only at the end

RUN python3 $WIKI_ROOT/faber_sync.py Drop and rebuild the index from wiki/**/*.md; regenerate index.md. Idempotent. write
03Telemetry · sources through the five phases

The pipeline, in motion.

Sources arrive at the left and move right through five named states. Most cross every node; some are rejected at DISCUSS on a skip directive; Phase 3b branches to record alignment before Sync. One sync at the end — every time.

A source dot spawns at READ. It moves along cobalt edges through DISCUSS, EXTRACT, WRITE, and SYNC. The ALIGN branch fires on most items — and increments ALIGNMENTS. A small fraction of items are rejected at the DISCUSS node (Narcis's skip directive); those count nowhere. Only items that reach SYNC get tallied as SYNCED.

04 Five rules · do not negotiate

Hazards, not negotiables.

These are the boundaries the skill enforces on itself. They exist because the wiki is append-only knowledge that compounds over years — small lapses become permanent. Read them as constraints on what the skill may decide, not suggestions.

HAZARD · 01 · IMAGE POLICY

Default zero images. Hard cap two. Every candidate clears four gates.

The wiki is text-first. Most images are decorative noise that bloats storage and dilutes search. Only architecture diagrams, flowcharts, data visualizations, or org charts are even considered — and only if a two-sentence description couldn't carry the same information.

If a candidate passes all four gates, present it to Narcis with pass/fail reasoning per gate. Wait for explicit approval. Save to wiki/assets/{slug}/ with a descriptive kebab-case filename. Hard cap: 2 per ingest.

REF · SKILL.md Phase 2b · FABER.md "Image Policy"
HAZARD · 02 · STANCE AUTONOMY

Never auto-create stances. Surface candidates only.

Stances are Narcis's to declare. The skill watches for claims that crystallize a position he hasn't yet stated, collects them in a list, and presents them in Phase 5 with (y/N/edit). Either way, log the outcome.

REF · Phase 3b · Stance candidate detection
HAZARD · 03 · APPEND, DO NOT OVERWRITE

Existing pages get merged, not replaced.

Read existing pages first. Add the new source, update connections, append new info — never overwrite. The page accumulates evidence over time; that accumulation is the compound.

REF · Phase 3 · entity/concept page updates
HAZARD · 04 · ALIGNMENT DISCIPLINE

Only non-neutral relations get written.

neutral is the default and pollutes the table. Don't emit it. Exception: if a page was previously non-neutral and this ingest flips it to neutral, write the correction. Be explicit about which claim moved the needle.

REF · Phase 3b · alignment persistence
HAZARD · 05 · SYNC ALWAYS LAST

End with faber-sync. Never skip.

Idempotent — always drops and recreates. Without it, the index disagrees with the markdown and every future probe lies. Run python3 $WIKI_ROOT/faber_sync.py as the final step.

REF · Phase 4 · update meta · rule list
05Operations · the five phases

Five phases. Five.

OP · A · PHASE 1

Read & discuss.

Read the source in full. Present 5–8 bullet points of key takeaways. Identify preliminary entities and concepts. Then ask Narcis what to emphasize and what to skip — and wait for direction before proceeding.

  • Surface pillar tensions explicitly during the summary, not silently
  • Preserve Romglish in Narcis's own quotes — the voice_rules govern
  • No extraction yet — read, summarize, ask
phase 1 · context loadingBASH
# Pull active pillars and stances first — the lens.
$ sqlite3 -json "$WIKI_ROOT/faber.db" \
    "SELECT pillars_json, stances_json FROM v_self_active_context;"

# Then read the source — file, URL, or pasted text.
$ cat "$1"    # or WebFetch URL

# Present 5–8 takeaways, then wait.
OP · B · PHASE 2

Extract entities, concepts, claims.

For each entity (people, companies, tools, frameworks): name, category, aliases. For each concept (patterns, models, principles): name, category, related concepts. For each notable claim: assertion plus a confidence read. Query faber.db for related pages first; note contradictions.

  • Entities get no alignment by default — exception: when the entity itself embodies a stance
  • Sources get aggregated alignment — reinforces X and weakens Y both land
  • Phase 2b — image evaluation — runs here. Default zero. HAZARD · 01
phase 2 · check before writeBASH
# Has this source landed before?
$ sqlite3 "$WIKI_ROOT/faber.db" \
    "SELECT slug, title FROM pages
     WHERE type='source'
       AND (source_ref LIKE '%kw%' OR title LIKE '%kw%');"

# Does this entity hide under another name?
$ sqlite3 "$WIKI_ROOT/faber.db" \
    "SELECT entity_slug, alias FROM aliases
     WHERE alias LIKE '%name%';"
OP · C · PHASE 3

Write pages. Append, never overwrite.

Source page in wiki/sources/{slug}.md with full frontmatter and prose summary. For each entity and concept: if the page exists, add the new source, update connections, append new info. If it doesn't, create it. Cross-references update across all affected pages. A synthesis page only when the source creates a significant new connection.

  • Kebab-case slugs · all filenames
  • YAML frontmatter must validate against FABER.md schema
  • Cross-link to vault docs via [[vault-path|Display]]
  • Conservative confidence at first — seed / medium

Align, then surface stance candidates.

The persisted judgment from the Phase 1 read. Write alignment: frontmatter on the new source page and on each new concept / synthesis (and on any page whose relevance to a pillar just shifted). Compare claims against stances_json; collect candidates that crystallize a new position or sharpen an existing one. Surface — do not auto-write.

07Conduct · how the skill carries itself

Speak to the wiki, not at it.

A

Read FABER.md first. Always.

Conventions, page types, schema — the source of truth lives in wiki/FABER.md. Don't infer from existing pages; they may pre-date the current schema.

B

Respect skip directives.

If Narcis says skip, skip. The Phase 1 discussion exists exactly so this can happen before any page is touched. Silent re-inclusion is a trust violation.

C

Start conservative.

When in doubt about maturity or confidence, default to seed or medium. Pages mature with evidence, not with optimism.

D

Cross-link both ways.

Wiki pages link to vault docs where the entity or concept is relevant — [[vault-path|Display Name]] — and vault docs gain wikilinks back. The graph is what makes the knowledge queryable.

Sources are ingested once · cross-referenced · synthesized. Knowledge compounds.
